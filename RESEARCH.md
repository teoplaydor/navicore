# NaviCore — Ресёрч рынка webcam-трекинга взгляда / головы / глаз / жестов

> Глубокое исследование под конкретную задачу: **портативная Windows-тулза**, которая через
> обычную веб-камеру отслеживает взгляд, положение головы, по-глазное моргание/подмигивание и
> жесты руки, работает в реальном времени на слабом железе без GPU, запаковывается в zip и
> запускается на любом компе. Ключевая механика: «подмигнул одним глазом на окне → повернул
> голову на другой монитор → открыл глаз → окно переехало в заранее заданную зону».
>
> Дата: 2026-06-09. Методология: многоагентный веб-ресёрч (15 агентов, ~800k токенов),
> проверка цифр по первоисточникам (GitHub README, офиц. доки, рецензируемые статьи, прайсы).
> Маркетинговые заявления помечены как скептические.

**Главный вывод-предостережение.** Точность взгляда обычной веб-камеры — **~3–4° ≈ 3–4 см на
экране** (правило: 1° ≈ 1 см на расстоянии ~57 см). Этого хватает, чтобы понять, **в какой
четверти/крупной плитке** экрана смотрит человек, но **не** какое именно (тем более
перекрывающееся) окно или какой точный пиксель. Поэтому вашу механику надо строить «наоборот»
относительно наивной идеи:

- **источник** окна = текущее активное окно ОС (`GetForegroundWindow`), взгляд — лишь грубое
  подтверждение региона;
- **цель** (какой монитор + какая зона) = **поворот головы** (крупный, медленный, малошумный
  сигнал) + предопределённые **большие зоны** (сетка 3×2 → «правая верхняя шестая»);
- **триггер** «захват/отпускание» = подмигивание одним глазом с **удержанием ~1 с** и отсевом
  обычного моргания.

Именно эту «инвертированную» схему вы и нащупали в постановке — она реализуема. Это
взаимодействие в стиле «крупная цель + осознанный жест», а **не** точная «глазная мышь».

## 1. Реальность точности webcam-взгляда (почему таргетинг — на голове, а не на взгляде)

Выверенные опорные цифры:

| Метод/инструмент | Точность (независимая) | Примечание |
|---|---|---|
| WebGazer.js | **4.17° / ~130 px / 4.06 см** (IJCAI-2016, Frontiers/PMC 2024) | дрейф **+49%** за 20 мин |
| WebEyeTrack/BlazeGaze | 2.32 см (GazeCapture), 4.56 см (MPIIFaceGaze) | дрейф всего **+20%** за 20 мин |
| GazeRecorder (лучший webcam) | **1.75 см только при зафиксированной голове**, −9% при движении | закрытый, .msi-инсталлятор |
| Tobii Eye Tracker 5 (железо, ИК) | **0.9° / 15 мм** | требует покупки железа |
| Pupil Core (ИК, наголовная камера) | sub-1° | не про экранный webcam |
| L2CS-Net (gaze-вектор) | 3.92° MPIIGaze (в пределах датасета) | ResNet50, не CPU-real-time |
| MobileGaze (малые бэкбоны, ONNX) | 12–13° MAE (Gaze360-only веса) | нужна дотренировка на MPIIFaceGaze до ~4–5° |

Скептические заметки к маркетингу: «sub-1.0°, как SMI RED 250» (GazeFlow) и «1.1–1.4°» у
WebGazer — это **best-case с зафиксированной головой**. Независимо подтверждённый потолок
обычной веб-камеры — порядка **3–4°**.

Три «убийцы» точности взгляда: **дрейф во времени** (нужна периодическая рекалибровка),
**движение головы** (2D-методы резко деградируют), **очки/блики/освещение/дистанция**.

**Почему голова надёжнее взгляда для выбора монитора/зоны:**
1. *Сигнал vs шум*: поворот головы к боковому/верхнему монитору — это десятки градусов,
   медленно и осознанно; ошибка позы головы у моделей 3–6° (6DRepNet 3.47° на AFLW2000).
2. *Трансляция*: «какой монитор» требует ещё и положения головы, не только направления глаз —
   матрица трансформации лица MediaPipe и 6DoF в opentrack дают и поворот, и позицию.
3. *Стабильность*: поза головы с фильтром (One-Euro/Kalman) стабильна всю сессию, взгляд
   дрейфует. Коммерческий Eyeware Beam свой мульти-мониторный указатель строит **именно на позе
   головы** — подтверждение подхода.

## 2. Рекомендованная архитектура (CPU-only, портативно)

```
            ┌─────────────────────── одна веб-камера @ 640×480, ~30 fps ──────────────────────┐
            │                                                                                 │
   OpenCV VideoCapture(CAP_DSHOW)                                                             │
            │                                                                                 │
            ▼                                                                                 ▼
  MediaPipe Face Landmarker (~3.7 МБ, Apache-2.0)                      MediaPipe Gesture Recognizer
   • 478 лендмарков + iris                                              (8.37 МБ, 8 жестов, ~17 мс CPU)
   • facial transformation matrix → ПОЗА ГОЛОВЫ (поворот+трансляция)             │
   • blendshapes eyeBlinkLeft / eyeBlinkRight (0..1) → ПО-ГЛАЗНОЕ МОРГАНИЕ        ▼
   • iris-vs-corner ratio → ГРУБЫЙ взгляд (только регион)                  жест → дебаунс → хоткей
            │                                                                (pynput / RegisterHotKey)
            ├──→ One-Euro фильтр (сглаживание позы/взгляда)
            │
            ▼
   КОНЕЧНЫЙ АВТОМАТ «захват → перенос → отпускание»
   ────────────────────────────────────────────────
   IDLE → (подмигивание одним глазом, удержание ~1 с, отсев бинокулярного моргания)
        → GRABBED: фиксируем GetForegroundWindow() как «источник»
        → пока глаз закрыт: поза головы выбирает целевой МОНИТОР + грубую ЗОНУ (сетка)
        → (глаз открылся, удержание открытого N кадров)
        → DROP: SetWindowPos(источник → rect зоны на целевом мониторе)
   + авто-сброс по таймауту (окно не «зависает», если глаз закрыт слишком долго)

   Win32 слой: GetForegroundWindow • EnumDisplayMonitors/GetMonitorInfo (virtual-desktop coords)
               • Per-Monitor-V2 DPI • SetWindowPos/MoveWindow
   UI слой:    трей (pystray/NotifyIcon) + always-on-top overlay (зоны/состояние) + RegisterHotKey
   Калибровка: по-мониторно, точки → ридж-регрессия в нормализованные 0..1 → virtual-desktop
```

**Стек прототипа:** Python 3.11–3.14 + MediaPipe Tasks (CPU/XNNPACK) + OpenCV + pywin32 +
pystray + pynput + Tkinter overlay. Однопроцессный, без GPU.
**Альтернатива для минимального размера в проде:** Tauri + WebView2 + MediaPipe Tasks-Vision
(WASM) + WebEyeTrack (MIT) + тонкий Rust/Win32-бэкенд (бандл <10 МБ против ~200–400 МБ у
PyInstaller). Нативный .NET (WinForms/WPF) — лучший по RAM/старту и доступу к Win32 API, если
готовы писать с нуля.

## 5. Сравнение: МОРГАНИЕ / ПОДМИГИВАНИЕ (по-глазно)

**Главное: проблема не в модели, а в логике «подмигивание vs моргание».** На любом
подмигивании EAR падает у ОБОИХ глаз ~на 40%, поэтому абсолютный порог по одному глазу даёт
ложные срабатывания на обычном (бинокулярном) моргании.

| Метод | Сигнал по-глазно | Размер/лицензия | Вердикт |
|---|---|---|---|
| **MediaPipe blendshapes `eyeBlinkLeft`/`eyeBlinkRight`** (0..1) | да, обученный | ~2 МБ / Apache-2.0 | **основной сигнал** подмигивания; порог «закрыт» >~0.5 |
| **EAR на лендмарках MediaPipe** | да, геометрический | 0 / Apache-2.0 | **кросс-проверка** к blendshapes; порог 0.15–0.3 |
| EAR на dlib-68 | да | модель ~95–100 МБ / Boost (но .dat — research-only) | формула хороша, модель dlib — нет (тяжесть+лицензия) |
| **Tracky Mouse (1j01)** | **готовый wink-to-click на MediaPipe FaceMesh, Windows** | MIT | **лучший референс кода** — форкнуть пороговую логику |
| OpenSeeFace LeftOpen/RightOpen | да | BSD-2 | готовая альтернатива, если берёте OSF как движок |
| Kalidokit | smoothBlink + `stabilizeBlink({noWink})` | MIT | только для JS/Electron-билда |
| Project IRIS / EVA Facial Mouse / eViacam | — | требует Tobii / Android / GPL | только UX-референсы |

**Надёжное правило подмигивания (для прототипа):**
```
wink = (blinkOne − blinkOther) > 0.35   AND   blinkOne > 0.5   AND   blinkOther < 0.3
arm  = wink удерживается ~24–30 кадров (≈0.8–1.0 с)   # естественное моргание длится 0.1–0.4 с
reject_bilateral = если оба глаза перешли порог в одном коротком окне → это моргание, игнор
hysteresis = разные пороги входа/выхода (анти-дребезг)
drop = глаз открылся и держится открытым несколько кадров
safety = авто-сброс захвата по таймауту, если глаз закрыт слишком долго
```
**Риск доступности:** часть людей не умеет подмигивать одним глазом. Обязательно нужен
**альтернативный триггер** (dwell / клавиша / жест).

## 7. Калибровка и мульти-монитор

- **Универсальный рецепт:** показать N точек → собрать пары «фичи глаза → координата экрана» →
  ридж-регрессия (WebGazer: 120-мерн. фича = патч 6×10 px на глаз; GazeRecorder: ~30 точек +
  отдельная калибровка лица; EyeGestures: 20 кругов → нормализ. 0..1; WebEyeTrack: few-shot, от
  9 сэмплов). Больше точек — точнее.
- **Мульти-монитор:** камера НЕ знает физической раскладки. Практика: калибровать в
  **нормализованных 0..1 на каждый монитор**, мапить в **virtual-desktop** координаты Win32,
  активный монитор выбирать сравнением луча головы/взгляда с прямоугольником каждого монитора
  (`GetMonitorInfo`). Вертикальные/повёрнутые мониторы — только по-мониторная калибровка +
  верная геометрия + **Per-Monitor-V2 DPI**.
- **Сглаживание обязательно:** 1-Euro (дёшево по CPU) или Kalman. Сырой кадр слишком шумный.
- **Дрейф:** для сессий длиннее пары минут — периодическая рекалибровка или head-pose-aware
  модель.
- **Failure modes (честно пользователю):** движение головы, неравномерный свет, очки/блики,
  дистанция, широкий FOV камеры (90–120° даёт overshoot; идеал 50–80°).

## 8. Упаковка под Windows + интеграция в ОС

| Тема | Рекомендация | Заметки |
|---|---|---|
| **Упаковка (Python)** | **PyInstaller `--onedir`** (zip-and-run) | избегать `--onefile`/Nuitka-onefile (распаковка в %TEMP%, AV-флаги). Самый «прозрачный» вариант — официальный **embeddable-Python ZIP** + вендоренные wheels (паттерн ComfyUI `python_embeded`) |
| **Упаковка (если C#)** | self-contained .NET single-folder | минимум RAM, быстрый старт, родной доступ ко всем Win32 API |
| **UI-шелл** | Tauri (~8.6 МБ, 30–50 МБ RAM) | Electron 80–244 МБ / 150–300 МБ RAM — конкурирует с CV по ресурсам, избегать. У Tauri нюанс: WebView2 может отсутствовать на Win10 — бандлить Evergreen runtime |
| **Захват камеры** | `cv2.VideoCapture(0, cv2.CAP_DSHOW)` | у MSMF-бэкенда баг медленного open/первого кадра. Доступ к камере **неэксклюзивный** |
| **Перемещение окон** | `SetWindowPos`/`MoveWindow` + `EnumDisplayMonitors`/`GetMonitorInfo` | ⚠️ `SetWindowPos` — это **запрос**: чужие окна могут отказать/зажать размер (та же проблема у FancyZones). Свои окна снапаются надёжно |
| **DPI** | объявить **Per-Monitor-V2** (манифест или `SetProcessDpiAwarenessContext` до создания окон), читать `GetDpiForMonitor` | иначе ОС битмап-масштабирует и врёт координаты → зоны промахиваются на hi-DPI/втором мониторе |
| **FancyZones как референс** | **MIT** (весь PowerToys), C++, снап через `SizeWindowToRect→SetWindowPos`, layout в `custom-layouts.json` | можно легально переиспользовать паттерны зон |
| **«Контекстное меню» портативно** | ⚠️ Explorer-пункт на Win11 требует COM `IExplorerCommand` + Sparse Package (НЕ zip-and-run). Легаси `HKCU\Software\Classes` — только под «Показать доп. параметры» | **для портативности — делать меню в приложении: трей (`NotifyIcon`/`pystray`) + overlay + `RegisterHotKey`** |
| **Трей/хоткеи/автозапуск** | .NET `NotifyIcon` (MIT) или `pystray` (LGPL-3.0, зависит от Pillow); глоб. хоткеи `RegisterHotKey`; автозапуск `HKCU\...\Run` или ярлык в `shell:startup` (без инсталлятора/админа, opt-in) | если нужен permissive-only стек — .NET-путь обходит LGPL pystray |
| **CPU real-time подтверждён** | OpenSeeFace 44–68 fps на 1 ядре; MediaPipe ~30 fps на CPU | модели — обычные файлы, полностью совместимо с zip-and-run + low-RAM |

> ⚠️ Важно про «контекстное меню» из постановки: настоящее меню правой кнопки в Проводнике
> ломает портативность (нужна регистрация COM/инсталлятор). Поэтому в прототипе меню/команды
> живут в **трее + overlay**, а глобальные действия — на **горячих клавишах**. Если позже нужен
> именно Explorer-пункт — это отдельный инсталлируемый режим, не zip-and-run.

## 11. Реализуемость вашей механики + риски + фоллбэки

**Вердикт:** РЕАЛИЗУЕМО при «инвертированном» дизайне (источник = foreground-окно + грубый
взгляд; цель = поза головы + большие зоны; триггер = подмигивание с удержанием). Всё на CPU,
20–30+ fps, пакуется портативно. Честно: это «крупная цель + осознанный жест», не точная глазная
мышь. Уверенность: средне-высокая (факты по точности/моделям/Win32/упаковке выверены; эмпирически
не проверены реальные false-accept/reject подмигивания и качество калибровки — мерить на целевом
железе).

**Топ-риски и митигации:**
1. Взгляд слишком груб (~3–4 см) → никогда не требовать пиксельного прицеливания взглядом;
   источник — `GetForegroundWindow`, цель — большие зоны.
2. Дрейф взгляда за сессию → head-pose-aware маппинг + периодическая рекалибровка.
3. Подмигивание ненадёжно/невозможно у части людей → дифф+удержание+отсев моргания +
   **альтернативный триггер** + авто-сброс по таймауту.
4. Движение головы/очки/свет/широкий FOV портят и взгляд, и blendshapes → подсказки на экране,
   сглаживание, graceful degradation.
5. Геометрия мониторов неизвестна камере → по-мониторная калибровка + реальные rect из
   `GetMonitorInfo` + Per-Monitor-V2 DPI.
6. Лицензионные ловушки → оставаться на Apache/MIT/BSD.
7. Нет опубликованного CPU-fps у части моделей → бенчить на целевом железе; MediaPipe как
   безопасный дефолт.

**Фоллбэки (по возрастанию надёжности):**
- F1: убрать экранный взгляд из таргетинга вовсе — и источник-регион, и цель брать с позы головы.
- F2: источник = только `GetForegroundWindow`; единственный пространственный вход — поза головы.
- F3: заменить подмигивание на dwell / жест руки / удержание модификатора.
- F4: взять готовый трекер (OpenSeeFace / opentrack) и писать только логику зон/перемещения.
- F5: для high-precision-юзеров — опц. поддержка Tobii 5 / TrackIR как «железного» входа.
- F6: целиком браузерный Tauri+WebView2 билд (WASM + WebEyeTrack) для бандла <10 МБ.
- Упрощение для де-риска: только snap-to-zone (без свободного пикселя), большие зоны (половины/
  трети/шестые), многосекундные удержания, явное подтверждение на экране перед перемещением.

### 13.1 Жесты: больше словарь + динамические (свайпы)

### 13.2 Оверлей: подсветка целевой зоны на реальном мониторе

**Рекомендация: tkinter-оверлей + ~15 строк pywin32-закалки. Ноль новых зависимостей.**

Механика (прослежена от исходников Tk до правила Windows): `-transparentcolor` у Tk
ставит WS_EX_LAYERED + SetLayeredWindowAttributes(LWA_COLORKEY) на wrapper-hwnd; по
документированному правилу хит-тестинга layered-окон цветовой ключ пропускает мышь
насквозь, а добавка WS_EX_TRANSPARENT (через SetWindowLongPtr на GetParent(winfo_id()))
делает кликабельно-прозрачным и нарисованное. Плюс WS_EX_NOACTIVATE|TOOLWINDOW (не
крадёт фокус, нет в таскбаре/Alt-Tab); показывать только через
SetWindowPos(HWND_TOPMOST, SWP_NOACTIVATE|...), не lift().

Дизайн: по одному withdrawn Toplevel на монитор (ректы уже есть из monitors.py, PMv2-DPI
уже выставлен); Canvas с ключевым цветом #010203; перерисовка ТОЛЬКО по событиям движка
(carry_started / zone_changed / arm_progress / dropped — единицы Гц), не по кадрам
камеры: SLWA-поверхность хранится системой, idle-CPU ~0%. UX-тайминги FancyZones
(сверено с их исходниками): fade-in 200 мс, вспышка на дропе 700 мс. Прогресс удержания
жеста/wink — дуга Canvas. Tk не thread-safe: события маршалить через
after()/event_generate. Опционально: живое превью переносимого окна в зоне через
DwmRegisterThumbnail (ctypes, дёшево — рендерит DWM; dest-hwnd должен принадлежать
нашему процессу). Анимировать ЧУЖОЕ окно (твин SetWindowPos) — не надо: дорого и
дёргано; анимируем контур на оверлее. FancyZones, к слову, делает прозрачность не через
WS_EX_LAYERED, а через DwmEnableBlurBehindWindow + Direct2D на отдельном потоке — для
Python это неподъёмно и не нужно. PySide6 ради оверлея (+77.5 МБ к бандлу) —
неоправданно.

### 13.3 Headless-режим (без debug-окна)

**Рекомендация: headless по умолчанию, debug-окно за флагом --debug.**

- cv2.waitKey без окна НЕ крутится вхолостую (замерено на бандле: блокирует 15–48 мс),
  но молча съедает клавиатурный ввод и латентность — из продакшн-пути убрать; цикл
  темпуется блокирующим cap.read() + наш FPS-кап.
- Статус: pystray проверен на бандле — HAS_NOTIFICATION=True (notify()), и Icon.icon —
  свойство с сеттером, т.е. живая смена иконки трея = индикатор состояния
  (idle/carrying/paused/stalled — пререндеренные PIL-иконки).
- Калибровка без окна: полупрозрачные topmost tkinter-подсказки на каждом мониторе
  (alpha ~0.85) + async-бипы winsound для подтверждений «без глаз». Тот же оверлейный
  стек, что и §13.2 — один механизм на двоих.
- Windows-Toasts и прочие toast-библиотеки — мимо: AUMID-регистрация в HKCU ломает
  zip-and-run; pystray.notify() хватает.
- opencv-python-headless почти ничего не экономит (GUI: WIN32UI, крошечный модуль);
  заметный мёртвый груз бандла — matplotlib+sounddevice, которые тянет metadata
  mediapipe (не выпиливаются штатно).

### 13.4 Шаринг камеры с другими приложениями

**Эмпирика на машине разработки (Win10 19045): NaviCore на CAP_DSHOW держит камеру
эксклюзивно — Discord одновременно с ним не получает картинку.** Замер: при занятой
камере DSHOW-open мгновенно падает (0.4 с), MSMF-open «открывается» ~9.3 с (известный
slow-open), но отдаёт 0 кадров. «Кто держит» видно в реестре
HKCU\...\CapabilityAccessManager\ConsentStore\webcam(\NonPackaged):
LastUsedTimeStop==0 → приложение использует камеру прямо сейчас (недокументировано,
работает с ~Win10 1903; код — defensively, try/except).

Проверенная картина по Windows: с Win10 1607 стандартный путь захвата (MF и DirectShow)
проксируется Camera Frame Server, но дефолтная семантика для десктопа — один
эксклюзивный клиент. Доступ «вторым» официально даёт ТОЛЬКО WinRT MediaCapture с
MediaCaptureSharingMode=**SharedReadOnly** (MS Learn, дословно: клиент «can receive
frames from the frame sources even if they are in use by another app»; Win10 14393+).
Колёса pywinrt (winrt-Windows.Media.Capture 3.2.1) под **cp314 win_amd64 есть** —
проверено прямым запросом PyPI JSON. Слух про жёсткий кап 640x480/NV12 в shared-режиме
первоисточниками НЕ подтверждается — зависит от драйвера, тестировать на месте.

Трёхслойная стратегия:
1. **Сейчас (низкая цена):** «супервизор сосуществования» на текущем CAP_DSHOW:
   ~15 подряд неудачных read() или серия чёрных кадров = камера потеряна → release +
   reopen с экспоненциальным backoff (с повторной установкой WxH/fps); плюс поллинг
   ConsentStore (~2 с): другой exe начал использовать камеру → проактивно отдать её
   (release, статус CAMERA YIELDED в трее), ушёл → вернуть.
2. **Настоящий фикс (средняя цена):** опциональный backend «shared camera» на WinRT
   MediaCapture SharedReadOnly via pywinrt: MediaFrameReader → Bgra8 SoftwareBitmap
   (MemoryPreference.Cpu) → numpy → существующий пайплайн; принимать формат
   контролирующего приложения; фоллбэк на слой 1 при неудаче инициализации.
3. **Документация:** Win11 24H2 имеет пер-камерный тумблер «Allow multiple apps to use
   camera at the same time» (подтверждено Insider-блогом, build 26120.2702); OBS Virtual
   Camera (GPL-2.0, мультиридер) — пользовательский обходной путь на Win10 (требует
   разовой админ-регистрации COM — в наш zip не встраивается).

MSMF-замечание: OPENCV_VIDEOIO_MSMF_ENABLE_HW_TRANSFORMS=0 чинит медленный open/первый
кадр (issue закрыт 2025-12-03), но шаринга НЕ даёт — Frame Server всё равно один
эксклюзивный клиент для обычного MF-пути.

### 13.5 Взгляд (повторно, коротко)

Скепсис обоснован (см. §1): webcam-взгляд = ~3–4° ≈ 3–4 см ≈ «квадрант экрана», с

### 13.8 «Одна камера → несколько ПРОИЗВОЛЬНЫХ приложений без виртуалки» на Win10 — НЕЛЬЗЯ

Прицельный адверсариально-проверенный ресёрч (источники — MS Learn, дословно). Краткий
ответ: **на Windows 10 это архитектурно невозможно** в портативном user-mode приложении.

**Почему (первоисточник).** Захват в Win10 идёт через Media Foundation Frame Server, и он
по умолчанию даёт камеру только одному «controlling»-клиенту:
> *«Only one controlling mode instance of IMFMediaSource can be in an active state at any
> time… will result in a sharing violation.»* (MF_DEVSOURCE_ATTRIBUTE_FRAMESERVER_SHARE_MODE)

Zoom/Discord/браузеры всегда открывают камеру в controlling-режиме (им нужно менять
формат/настройки), и перекомпилировать их мы не можем → второй получает sharing violation.
Заставить чужое приложение перейти в sharing-режим из вне — нельзя.

| Подход | vcam | драйвер | произвольные апп. | вердикт |
|---|---|---|---|---|
| Frame Server по умолчанию | нет | нет | нет | один controlling-клиент за раз |
| **WinRT MediaCapture SharedReadOnly** (наше приложение читает, пока чужое контролирует) | нет | нет | только своё | **РАБОТАЕТ на Win10 1607+** (`MediaCaptureSharingMode`, since 10.0.14393) |
| Win32 MF share-mode атрибут | нет | нет | нет | **Win11 26100+ только** |
| Win11 24H2 тумблер «Multi-App Camera» | нет | нет | да | **Win11 only**, нет порта/реестра/политики на Win10; не упаковать |
| DShow Bridge / EnableDshowRedirection | нет | да (IHV/HKLM по VID&PID) | только cooperating | не портативно, не для чужих камер |
| Все реальные «сплиттеры» (SplitCam, ManyCam, e2eSoft, OBS, XSplit) | **да** | **да** | да | именно то, что исключено: ставят виртуалку+подписанный драйвер, апп выбирает ВИРТУАЛЬНУЮ камеру |
| DLL-инъекция/хук в Zoom/Discord | нет | нет | нет | защищённые процессы, античиты, антивирусы — тупик |

**Что реально можно (портативно, без драйвера, на Win10):** наше приложение открывает
камеру в **SharedReadOnly** (WinRT MediaCapture) и читает кадры, **пока ровно одно**
другое (controlling) приложение — например, Zoom — её держит. Это даёт «NaviCore + один
обычный апп» одновременно, без установки/админа/драйвера. Чего это НЕ даёт: два чужих
controlling-приложения (Zoom И Discord) одновременно — для этого нужен либо Win11 24H2 с
тумблером, либо исключённая категория «виртуалка + драйвер».

**Поправка к пулу:** raw Win32 MF-атрибут — действительно только Win11 26100+, НО WinRT-енум
`MediaCaptureSharingMode.SharedReadOnly` существует на Win10 с 1607 — это и есть лазейка для
нашего (кооперирующего) кода. Колёса pywinrt под cp314 проверены ранее (§13.4).
